# TASK: Implement Tenant-Aware Query Adapters

---

## DESCRIPTION

- Summary (1 sentence): Add query-level scoping so tenant-bound data cannot leak through list or lookup operations.
- Steps (max 5):
  1. Identify repositories and use cases that return tenant-sensitive data.
  2. Introduce query adapter patterns or scoped repository methods that require tenant context.
  3. Apply tenant filters consistently in list and search operations.
  4. Make tenant-scoped access explicit in read paths and joins.
  5. Document the expected filtering strategy for sensitive repositories.
- Implementation hints:
  - Route guards are not enough; the repository layer must also prevent leakage.
  - Prefer explicit scoped methods over optional parameters that can be forgotten.
  - Use the request authorization context to build the query scope.
  - All tenant-sensitive repositories MUST expose explicit tenant-scoped methods (e.g., `findAllByTenant(tenantId)`).
  - Optional tenant parameters are NOT allowed for sensitive queries.
  - The repository contract must make it impossible to forget tenant scoping.
  - Tenant users MUST NEVER control tenantId through request parameters.
  - Tenant scope must always be derived from authenticated context.
  - Platform users MAY provide tenantId as a filter, but it must be explicitly validated.
